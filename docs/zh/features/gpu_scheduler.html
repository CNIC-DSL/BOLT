GPU 并发调度

bolt-lab 的并发模型围绕 GPU 令牌池构建：任务启动前获取一个令牌，任务完成后归还令牌。这控制了并发级别和 GPU 绑定行为。

1. 关键参数

  • run.gpus
    指定可用 GPU 列表，例如 [0,1,2]。

  • run.slots_per_gpu（默认：1）
    每个 GPU 提供多少个令牌。slots_per_gpu=2 表示允许同一个 GPU 同时运行 2 个任务（适用于小模型/小 batch size 场景）。

  • run.max_workers
    最大并发工作线程数。实际并发数为： effective_concurrency = min(max_workers, total_GPU_tokens, number_of_combos)

2. 运行时行为

  • 组合启动前，从令牌池获取一个 gpu_id。

  • 子进程启动时，设置 CUDA_VISIBLE_DEVICES 以绑定到获取的 gpu_id。

  • 组合完成后，归还令牌，以便后续组合复用。

该机制具有以下特性：

  • 并发数有上限，防止无限并行导致 GPU 显存耗尽

  • 使用多个 GPU 时，任务会自然分布到不同的 GPU 上

  • slots_per_gpu 可在显存允许时提高吞吐量

仅 CPU 回退模式

当 gpus 为空（即 gpus: []）时，框架回退到仅 CPU 模式：使用一个 None 令牌，提供一个槽位且不绑定 GPU。在此模式下，不设置 CUDA_VISIBLE_DEVICES，子进程在 CPU 上运行。这适用于不需要 GPU 资源的方法或快速调试。

3. 推荐配置示例

  1. 单 GPU，严格串行（最稳定） run: gpus: [0] max_workers: 1 slots_per_gpu: 1

  2. 多 GPU，每个 GPU 一个任务（常用默认配置） run: gpus: [0,1,2,3] max_workers: 4 slots_per_gpu: 1

  3. 单 GPU 多任务（仅在显存充足时使用） run: gpus: [0] max_workers: 2 slots_per_gpu: 2

4. 启用并发前先用 dry_run 检查

首次配置并发时，建议先设置 dry_run=true：

  • 仅打印将要执行的命令和参数

  • 帮助确认方法入口点、数据集、fold/seed 组合等是否符合预期

  • 避免在启动大量组合后才发现配置错误
