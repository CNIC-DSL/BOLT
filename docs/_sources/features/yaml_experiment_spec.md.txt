# YAML Experiment Specification

In bolt-lab, a batch experiment is fully described by a single YAML file. This YAML mainly consists of 8 parts: `methods`, `datasets`, `result_file`, `grid`, `run`, `paths`, `per_method_extra`, and `extra_args`.

This page explains what these fields mean, how to set their values, and provides a minimal example you can reuse directly.

## 1. Top-Level Fields Explained

1) `methods`  
The list of methods to run. The framework resolves each method to a task (`gcd` or `openset`) using a built-in lookup table (`_METHOD_TASK_MAP` in `configs/base.py`) and runs them across the combinations generated by `grid`.

2) `datasets`  
The list of datasets to run. Each dataset is combined with `methods` and `grid` to form the final experiment combos.

3) `result_file`  
A naming suffix for summary files. Under `results_dir`, the framework will generate:
- `summary_{result_file}.csv`

4) `grid`  
The grid search space. The framework takes the Cartesian product across all dimensions in `grid` (via `itertools.product` in `GridConfig.iter_experiments()` in `configs/base.py`) to generate all combos.
Common dimensions include:
- `known_cls_ratio`: ratio of known classes
- `labeled_ratio`: ratio of labeled data
- `fold_types / fold_nums / fold_idxs`: cross-validation split type and indices. Valid values for `fold_types` are `fold` and `imbalance_fold`.
- `seeds`: random seeds
- `cluster_num_factor`: cluster-number multiplier (if a method uses this parameter)

5) `run`  
Runtime control parameters, commonly including:
- `gpus`: list of available GPUs (e.g., `[0,1,2]`)
- `max_workers`: maximum number of concurrent workers
- `slots_per_gpu`: number of concurrent "slots" allowed per GPU (default: 1)
- `dry_run`: print commands only, do not execute (default: false)
- `retry_on_oom`: retry on OOM or not (default: true)
- `max_retries`: maximum number of OOM retries (default: 2)
- `retry_backoff_sec`: seconds to wait before each retry (default: 15)

It also includes training epochs:
- `num_pretrain_epochs`
- `num_train_epochs`

6) `paths`  
Output path settings:
- `results_dir`: directory for summary files (e.g., `results`)
- `logs_dir`: directory for logs (e.g., `logs`)

7) `per_method_extra`  
Per-method parameter overrides/additions. This is used to inject extra parameters into a specific method's runtime arguments, for example:
- specify `backbone`, `reg_loss`, `extra_flags`, etc. for a specific method
- specify `llm_model`, `llm_api_base`, etc. for OpenSet LLM-based methods

8) `extra_args`  
An optional top-level field for global extra arguments that apply to all methods. These are merged into every experiment combo's arguments before per-method overrides are applied.

## 2. Minimal Example (Copy-Paste Ready)

The following example shows a minimal runnable YAML structure. For the first run, it is recommended to shrink `methods` / `datasets` / `grid` to the smallest scale to validate the pipeline.

```yaml
methods: [tan]
datasets: [X-Topic]
result_file: demo

grid:
  known_cls_ratio: [0.75]
  labeled_ratio: [0.5]
  fold_types: [fold]
  fold_nums: [5]
  fold_idxs: [0]
  seeds: [2025]
  cluster_num_factor: [1.0]

run:
  gpus: [0]
  max_workers: 1
  slots_per_gpu: 1
  num_pretrain_epochs: 1
  num_train_epochs: 1
  dry_run: false

paths:
  results_dir: results
  logs_dir: logs

per_method_extra: {}
extra_args: {}
```

## 3. How to Compute the Number of Combos

number_of_combos = (number of methods) × (number of datasets) ×
the Cartesian-product size of:
`known_cls_ratio × labeled_ratio × fold_types × fold_nums × fold_idxs × seeds × cluster_num_factor`.

When the combo count is large, it is recommended to use `dry_run` first to check whether commands and parameters match your expectations, then gradually scale up the grid.

## 4. Example of Using `per_method_extra`
```yaml
per_method_extra:
  plm_ood-llm:
    llm_ood: true
    llm_model: gpt-4.1-mini
    llm_temperature: 0.0
    llm_batch_size: 16
  plm_ood:
    reg_loss: cosine
  ab:
    emb_name: sbert
    extra_flags: ["--some_flag", "1"]
```
