# Core Concepts

This page provides unified explanations of key terms that repeatedly appear in the **bolt-lab** documentation, so you can quickly understand configurations and outputs.

## 1. Task: `gcd` and `openset`

bolt-lab currently supports two types of tasks:
- `gcd`: Generalized Category Discovery
- `openset`: Open-set Recognition / Open-set Classification

In the configuration files, methods are grouped under a specific task. The framework uses this to select the corresponding execution entry point and metric/result fields.

## 2. Method: Algorithms (Implementations)

A `method` is the name of the algorithm you want to run, such as `tan`, `ab`, `plm_ood`, etc.
The same method can be run multiple times under different datasets, splits, and hyperparameter combinations.

The relationship between methods and tasks is handled automatically by the framework's internal lookup table (`_METHOD_TASK_MAP` in `configs/base.py`). You do not need to specify any `maps` field in YAML.

## 3. Dataset

A `dataset` is a dataset identifier (a string).
The framework expands the Cartesian product of `methods × datasets × grid` into experiment combinations and schedules them one by one.

## 4. Grid: Hyperparameter Search Space

A `grid` describes which parameter dimensions to sweep. Common dimensions include:
- `known_cls_ratio`: ratio of known classes
- `labeled_ratio`: ratio of labeled data
- `fold_types / fold_nums / fold_idxs`: data split strategy and indices
- `seeds`: random seeds
- `cluster_num_factor`: cluster-number multiplier (used by some methods)

Each dimension is a list, and the framework takes the Cartesian product across all dimensions.

## 5. Combo: An Experiment Combination (Minimal Execution Unit)

A `combo` is the minimal execution unit, typically uniquely determined by:
- task, method, dataset
- `known_cls_ratio`, `labeled_ratio`, `cluster_num_factor`
- `fold_type`, `fold_num`, `fold_idx`
- seed
- training epochs (`num_pretrain_epochs`, `num_train_epochs`)
- additional parameters injected via `per_method_extra`

For each combo, the framework produces logs and output artifacts, and collects results into summary tables.

## 6. Stage: Multi-Stage Pipeline (Per Method)

Some methods consist of multiple stages (e.g., pretraining followed by finetuning). Each stage is called a `stage`.
The framework executes `stage1`, `stage2`, ... in order. If any stage fails, that combo fails and subsequent stages will not run.

## 7. Workdir: Working Directory (`--output-dir`)

You specify the working directory via `--output-dir` at runtime.
The framework organizes outputs such as `logs/`, `results/`, and `outputs/` under this directory, and also keeps configuration copies for the run to support reproducibility and archiving.

## 8. Summary: Summary Table (`summary_{result_file}.csv`)

After each combo finishes, the framework attempts to collect the latest result record for that method and appends it to the summary table:
results/summary_{result_file}.csv

The summary table is the preferred entry point for subsequent comparison and analysis.
